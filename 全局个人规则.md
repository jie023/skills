## 规则优先级与适用范围

- 用户当前对话中的明确要求优先级最高

## Skill 触发索引

- 涉及中文文件、Windows 中文路径、UTF-8、乱码、编码敏感写入：使用 `windows-utf8-safe-editing`
- 涉及多文件修改、批量替换、全局格式化、迁移字段或文案：使用 `safe-bulk-editing`
- 涉及 Java 编写、修改、评审、中文 JavaDoc、P3C、`@author`：使用 `java-personal-standards`
- 涉及用户输入、SQL、命令执行、权限、敏感数据、日志脱敏、XSS：使用 `secure-code-checklist`
- 仅当用户明确要求 Git 操作、提交或提交信息时：使用 `git-commit-helper`

## 一、代码规范

- 所有代码注释必须使用中文；代码、命令、API 名称和专有名词可保留原文
- Java 代码遵循 `java-personal-standards`，作者名取“基本信息”中的“作者”字段
- 优先遵循项目现有规范，优先复用项目已有工具类和公共方法
- 修改完成后优先运行相关测试、构建或静态检查；无法运行时说明原因、风险和替代复核

---

## 二、任务处理流程

### 2.1 任务确认流程
- **简单任务**（如查文件、改普通配置、单文件小改动）：直接执行
- **普通多文件代码修改**：先列出简要计划和影响范围，再按计划执行，无需额外等待确认
- **高风险任务**（架构变更、删除文件/数据、修改生产配置、安全敏感操作、强制 Git 操作）：必须先整理问题并复述确认，等待用户明确确认后再执行
- 不确定是否属于高风险任务时，主动询问用户，不要假设

### 2.2 任务规划（超过3个步骤的任务必须使用）
- 使用当前平台可用的任务清单工具创建和维护任务计划
- 任务状态必须及时更新为 `in_progress` / `completed`
- 任务完成后及时标记为完成

平台映射：
- Codex：使用 `update_plan`
- Claude：使用 `TodoWrite`、`TaskCreate`、`TaskUpdate` 等可用任务工具
- Trae：使用可用的任务清单工具

### 2.3 代码搜索优先级
优先使用当前平台可用的代码搜索工具：
1. 按内容或关键词搜索代码
2. 按文件名模式查找文件
3. 读取具体文件内容
4. 复杂搜索场景再使用子代理或探索型工具

平台映射：
- Codex：优先使用 `rg`、`rg --files`，再读取文件
- Claude：优先使用 `Grep`、`Glob`、`Read`
- Trae：优先使用 `SearchCodebase`、`Read`

**搜索结果必须包含：** 文件路径 + 具体代码内容 + 行号

## 三、交互规范

### 3.1 用户确认
**必须使用当前平台可用的用户确认方式确认的情况：**
- 删除文件、数据库、数据
- 修改生产环境配置
- 强制推送（git push --force）
- 涉及安全敏感操作（修改密码、密钥）

**无需确认的情况：**
- 修改普通代码文件
- 创建新文件
- 读取/搜索代码

不确定时主动询问，不要假设

### 3.2 回答风格
- **每次回复必须以【李杰】开头**
- **简单问题直接回答，不要过度展开**
- 保持回答简洁明了
- 涉及代码时给出具体文件路径和行号

---

## 四、错误处理

### 4.1 命令执行
- 命令执行失败时先分析原因
- **不要重复执行相同的失败命令**
- 分析失败原因后调整策略再执行

---

## 五、文件、编码与工具

- 工具中使用正斜杠 `/`，例如 `C:/Users/李杰/project`
- 读取/写入文件时优先使用绝对路径；中文或空格路径必须谨慎处理
- 涉及中文文件、UTF-8、乱码或编码敏感写入时，按 `windows-utf8-safe-editing` 执行
- 发现中文乱码时，不得基于乱码内容修改，必须先重新确认编码
- 命令工具仅用于系统命令、搜索、构建、测试等场景，不用于绕过编辑规范

---

## 六、依赖和联网规则

- 允许读取公开文档、本地依赖文件和本地配置用于分析
- 安装新依赖、升级或降级核心依赖、修改依赖锁文件前必须先确认
- 访问生产环境、预发布环境、需要登录或鉴权的远程资源前必须先确认
- 不得将密码、密钥、Token 等敏感信息发送到外部服务
- 网络请求失败时先分析原因；如需提升权限或重试，必须说明目的和范围

---

## 七、批量操作规则

- 批量读取文件可以同时进行；批量写入默认禁止并发执行
- 涉及多文件修改、批量替换、格式化或迁移时，按 `safe-bulk-editing` 执行
- 修改超过 3 个文件、需要脚本写入或可能影响编码/格式时，必须先列影响范围和风险点
- 未经确认，不得使用批量脚本或命令直接写文件

---

## 八、Git 操作限制

- **无明确要求时，不要执行或触碰任何 Git 操作**
- 包括但不限于：`git status`、`git diff`、`git add`、`git commit`、`git push`、`git checkout`、`git reset`
- 用户明确要求 Git 操作、提交或提交信息时，按 `git-commit-helper` 执行
- 强制推送、重置、覆盖本地文件、改写历史等高风险 Git 操作必须单独确认

---

## 九、代码安全规范

- 涉及用户输入、SQL、命令执行、路径、模板、权限、敏感数据、日志或前端输出时，按 `secure-code-checklist` 执行
- 禁止直接拼接用户输入到 SQL、命令、路径或模板
- 敏感数据禁止硬编码，日志和接口输出必须避免泄露完整敏感值
- 涉及认证、授权、权限变更、数据导出等操作时，必须进行身份校验和权限校验
