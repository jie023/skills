# 通用代码简化审查

## 定位

代码简化审查用于发现过度复杂、重复、分支过深、抽象过度或可读性差的代码。审查结论必须以保持现有行为不变为前提，只输出问题、证据和建议；除非用户明确要求修复，否则不自动改代码。

## 执行流程

1. 确认审查范围：优先使用用户指定文件、当前任务涉及文件或当前打开上下文；没有范围时先用平台搜索工具定位候选文件。
2. 读取待审查文件及相邻调用方、测试或配置，确认当前行为边界。
3. 建立行为基线：记录公开接口、输入输出、异常路径、副作用、异步顺序、事务边界和权限校验等不可改变点。
4. 搜索复杂度线索：用内容搜索找重复代码、深层分支、嵌套三元、调试残留、注释代码、单次使用抽象和长参数列表。
5. 逐项判断等级：每个发现必须有文件路径、行号、证据、影响和可保持行为不变的建议；证据不足时标为待确认，不强行定级。
6. 输出审查报告：按阻断、高风险、中风险、建议优化分组；没有问题时说明已检查范围、未发现项和剩余风险。
7. 用户要求修复时：只做行为等价的小步修改，优先沿用项目现有工具类、命名和架构；多文件修改按安全批量修改规则执行。

## 内置规则覆盖

- 简化原则：减少重复、降低嵌套、缩小函数、消除无意义抽象、让命名表达意图，但不得改变外部行为。
- 优先级：先处理能减少真实风险的复杂度，再处理可读性建议；不要为了“看起来简洁”牺牲清晰性。
- 行为等价：涉及异常语义、事务边界、权限校验、异步顺序、缓存和副作用时，必须先证明等价或标为待确认。
- 反模式：单次使用抽象、过度泛型、过深继承、重复 if/else、嵌套三元、长参数列表、注释掉的代码和调试残留。
- 输出建议：给出最小可行重构方向，例如提取函数、提前返回、合并重复分支、引入值对象或删除无效间接层。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。

## 检查项

| 等级 | 检查项 | 怎么检查 |
| --- | --- | --- |
| 阻断 | 简化建议会改变外部行为、数据结果、异常语义或副作用顺序 | 对照调用方、测试、公开接口和关键分支；只要不能证明等价，就不要给出直接改法，改为说明风险和需要补充的验证 |
| 阻断 | 重复分支或重叠条件导致关键路径不可达、结果冲突或必需副作用被跳过 | 逐行追踪 `if/else`、`switch/case`、早返回、异常抛出和状态写入；核对相同输入是否会落入冲突分支 |
| 阻断 | 删除“无用代码”前无法确认其是否承担兼容、注册、反射、序列化或框架约定职责 | 搜索引用、配置、注解、路由、模板、反射字符串和测试；无法确认时标待确认，不建议直接删除 |
| 高风险 | 深层嵌套、长方法或复杂布尔表达式混合状态修改、IO、事务或权限逻辑 | 统计嵌套层级和出口数量；标出每个分支的输入条件、状态变化和返回值，判断是否可用早返回或命名条件拆开 |
| 高风险 | 大段复制粘贴逻辑只有少数字段不同，后续维护容易遗漏 | 搜索相同函数调用、相同错误处理、相同 SQL/请求组装或相同 UI 分支；比较差异是否能由参数、局部 helper 或映射表表达 |
| 高风险 | 过度抽象的公共 helper 通过布尔参数、模式字符串或可选参数承载多种业务 | 搜索 helper 的所有调用点；检查参数组合、默认值和分支是否让调用方难以理解，必要时建议拆回清晰的局部函数 |
| 高风险 | 异步链、回调、锁、事务或资源释放流程可读性差 | 检查 `await/then/callback` 混用、异常传播、取消逻辑、finally/close/dispose、锁释放顺序；建议前必须确认时序不变 |
| 中风险 | 单次使用的包装函数、接口、常量或中间变量增加跳转成本 | 搜索引用数量和调用上下文；如果没有复用、隔离变化或表达业务含义的价值，建议内联或合并 |
| 中风险 | 命名含糊、临时变量过多或链式调用过长，导致意图不清 | 检查变量名是否表达业务含义；长链是否混合解析、校验、转换和副作用；建议拆成有意义的中间步骤 |
| 中风险 | 注释代码、调试输出、无用导入、无效 TODO 或过期说明影响阅读 | 搜索调试语句和注释块；确认不是项目约定、生成代码或临时诊断要求后再建议移除 |
| 建议优化 | 可以用早返回、卫语句、命名条件、局部 helper 或映射表降低认知负担 | 对照现有项目风格，给出最小可读性优化；不要为了“更短”牺牲明确性 |
| 建议优化 | 可以合并重复字面量、错误消息、字段映射或校验规则 | 搜索相同文本、枚举值、字段名和校验条件；建议复用已有常量、枚举或工具类 |
| 建议优化 | 可用项目已有工具类或语言标准库替代手写样板逻辑 | 先搜索项目现有工具和类似实现；只在语义完全一致时建议替换 |

## 常用搜索线索

优先使用当前平台可用搜索工具，并要求结果带文件路径、行号和命中内容。

```text
rg -n "TODO|FIXME|HACK|XXX|待优化|临时|兼容" <path>
rg -n "console\\.log|debugger|System\\.out\\.println|printStackTrace|print\\(" <path>
rg -n "\\bif\\b|\\belse\\b|switch|case|\\?|&&|\\|\\|" <path>
rg -n "then\\(|catch\\(|callback|async|await|finally|dispose|close\\(" <path>
rg -n "copy|clone|duplicate|common|helper|util|flag|mode|type" <path>
rg -n "return|throw|break|continue" <path>
```

搜索后不要只凭命中数量判断复杂度，必须阅读上下文并确认行为边界。

## 输出格式

有发现时按以下格式输出：

```markdown
## 阻断
- [阻断] path/to/file.ext:123 标题
  - 怎么检查：说明读取了哪些调用方、分支、测试或配置，命中了什么证据
  - 原因：为什么当前写法难以维护或存在行为风险
  - 建议：给出保持行为不变的最小处理方案
  - 验证：需要运行或补充的测试、构建、静态检查或人工确认点

## 高风险
...

## 中风险
...

## 建议优化
...
```

无发现时输出：

```markdown
未发现明确的代码简化问题。

已检查范围：
- path/to/file.ext: 重点检查了分支、重复逻辑、抽象层级和调试残留

剩余风险：
- 未运行测试或缺少某类上下文时，在这里说明
```

## 多平台兼容

- Codex：优先 `rg`、`rg --files` 搜索，读取文件时保持 UTF-8；修改普通代码优先 `apply_patch`，但本 skill 默认只审查不修改。
- Claude：优先 `Grep`、`Glob`、`Read` 定位和读取；需要修改时使用 `Edit` 或 `MultiEdit`，并保持中文 UTF-8。
- Trae：优先 `SearchCodebase`、`Read` 获取文件路径、行号和上下文；需要修改时使用平台编辑工具。
- 所有平台：不默认执行 Git 操作，不依赖外部旧目录，不启用 hooks、后台观察或自动修复流程；Windows 路径优先使用正斜杠。
