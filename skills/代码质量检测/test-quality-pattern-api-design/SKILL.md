# API 设计模式质量审查

## 定位

面向 REST API 设计和代码契约的静态质量审查。默认只读代码、路由、DTO、Controller、Service 边界、OpenAPI/Swagger 文档和测试，不跑 API 请求，不自动修复代码。

## 内置规则覆盖

- 资源建模：URL 使用资源名词和层级关系，非 CRUD 动作只在无法资源化时使用动作子资源。
- HTTP 语义：方法、状态码、幂等、缓存、条件更新和错误响应必须符合接口行为。
- 契约稳定：请求/响应 schema、错误码、分页、过滤、排序、版本和兼容性要有明确规则。
- 安全边界：鉴权、对象级授权、租户隔离、输入校验、字段白名单和敏感信息隐藏必须检查。
- 文档测试：OpenAPI、契约测试、Controller/Handler 测试、权限测试和兼容性回归是审查依据。
- 限流契约：公开 API 应明确调用方 tier、配额维度、`Retry-After`、`X-RateLimit-*` 响应头和超限错误体。

## 执行流程

1. 明确审查范围：确认用户提供的是接口、模块、变更文件还是全项目；范围不清时先说明假设。
2. 按内置规则读取接口定义、Controller/Router、DTO/schema、鉴权中间件、错误处理和调用方。
3. 搜索接口入口：定位路由、Controller、Handler、DTO、OpenAPI、鉴权中间件、错误处理和测试文件。
4. 逐项检查契约：按资源命名、HTTP 语义、响应结构、错误码、分页过滤排序、版本、幂等、鉴权、输入校验、文档、兼容性、测试进行审查。
5. 分级输出问题：按阻断、高风险、中风险、建议优化分组；每条必须包含文件路径、行号、问题、影响、依据和建议。
6. 明确不确定项：无法从静态代码确认的事项要说明缺口和下一步验证方式，不得把猜测写成事实。

## 分级标准

### 阻断

会导致接口不可安全发布、核心契约不可用或严重破坏兼容性的设计问题。

- 对外接口缺失认证或授权边界，敏感资源可被未授权访问。
- 删除、支付、审批、状态流转等高影响操作未设计幂等或重复提交保护。
- 已发布 API 删除字段、改字段类型、改 URL、改鉴权方式且没有新版本或兼容层。
- OpenAPI/接口文档与实现关键契约冲突，导致客户端无法正确调用。
- 错误响应泄露堆栈、SQL、密钥、内部服务地址等敏感信息。

### 高风险

会造成明显业务错误、安全风险、客户端大量适配成本或生产排障困难。

- URL 使用动词式动作替代资源模型，如 `/getUser`、`/deleteOrder`。
- HTTP 方法和状态码语义错误，如创建返回 200、校验失败返回 500、未鉴权返回 200。
- 列表接口缺失分页、排序稳定性或最大页大小限制，可能拖垮数据库或响应超大。
- 错误码不稳定、不唯一或没有字段级校验详情，客户端无法可靠处理。
- 资源级鉴权只校验登录状态，未校验租户、组织、所有者或角色权限。

### 中风险

会降低一致性、可维护性或长期演进能力，但通常可通过兼容改造逐步修复。

- 资源命名单复数、大小写、连接符风格不统一。
- 过滤、排序、搜索参数命名不一致，缺少白名单或边界约束。
- 响应 envelope、时间格式、ID 类型、空集合返回结构不一致。
- API 版本策略缺失，破坏性和非破坏性变更边界不清。
- 只有实现没有契约测试、Controller 测试或 OpenAPI 校验。

### 建议优化

不直接阻断发布，但能提升开发者体验、可观测性、兼容性或长期治理质量。

- 为创建接口补充 `Location` 响应头，为限流补充 `Retry-After` 和限流头。
- 为不同调用方 tier 补充配额规则、剩余次数、重置时间和超限后的错误码/响应头。
- 为公开 API 补充弃用说明、`Sunset` 头或迁移文档。
- 为列表接口补充游标分页、稀疏字段、include/expand 或统一搜索参数。
- 为错误响应补充 requestId/traceId，便于排查问题。
- 为 OpenAPI 增加示例、错误响应 schema、安全方案和分页 schema。

## 检查重点

- REST 资源命名
- HTTP 方法和状态码
- 请求、响应和错误结构
- 分页、过滤、排序、搜索
- 版本管理
- 幂等和重复提交保护
- 限流响应头和调用方配额层级
- 认证、授权和多租户隔离
- 输入校验和边界约束
- OpenAPI/Swagger/接口文档
- 向后兼容性和弃用策略
- 契约测试、Controller 测试和回归测试

## 具体检查方法

### REST 资源命名

- 资源路径优先使用名词、复数、小写、kebab-case；避免动词、单数、驼峰、下划线混用。
- 层级嵌套只表达真实从属关系，避免过深路径；跨资源查询优先使用 query 参数。
- 非 CRUD 动作只在确实无法建模为资源时使用，并优先使用 `POST /resources/{id}/actions` 类形式。
- 检查同一模块内 URL 前缀、版本前缀、ID 占位符命名是否一致。

### HTTP 方法和状态码

- `GET` 只读且安全；`POST` 创建资源或触发动作；`PUT` 完整替换；`PATCH` 局部更新；`DELETE` 删除资源且应按幂等语义处理。
- 创建成功优先 `201 Created`，并在需要时返回 `Location`；删除无响应体优先 `204 No Content`。
- 参数格式错误使用 `400`，语义校验失败可使用 `422`，未登录 `401`，无权限 `403`，资源不存在 `404`，冲突 `409`，限流 `429`。
- 禁止所有失败都返回 `200`，禁止业务错误滥用 `500`，禁止把内部异常细节直接暴露给客户端。

### 请求、响应和错误码

- 成功响应结构要稳定，公开 API 优先统一 `data/meta/links` 或项目既有 envelope。
- 错误响应至少包含稳定 `code`、可读 `message`；输入校验错误应包含字段路径、字段错误码和错误原因。
- 错误码应可枚举、可文档化、可兼容扩展；避免把中文文案、异常类名或数据库错误当作客户端判断依据。
- 时间、金额、枚举、ID、布尔值、空值、空数组的序列化规则必须在接口间一致。

### 分页、过滤、排序和搜索

- 列表接口必须有分页；大数据量或公开 API 优先游标分页，管理后台小数据量可使用 offset/page。
- 分页参数要有默认值、最大值、稳定排序字段和越界行为；游标应不透明，不能暴露可篡改内部状态。
- 过滤字段、排序字段、搜索字段必须白名单校验，避免任意字段拼接 SQL、NoSQL 或 ORM 表达式。
- 多字段排序、范围过滤、模糊搜索、include/expand、fields 等能力要和文档保持一致。

### 版本、兼容性和弃用

- 已发布 API 的破坏性变更必须通过新版本、兼容层或明确迁移策略处理。
- 非破坏性变更包括新增可选字段、新增可选参数、新增接口；删除字段、改类型、改必填、改鉴权、改 URL 属于破坏性变更。
- 公开 API 应有弃用时间线、兼容窗口和迁移说明；可使用 `Deprecation`、`Sunset`、`Link` 等响应头辅助治理。
- 检查 OpenAPI、客户端 SDK、前端调用和测试是否同步反映版本变化。

### 幂等和并发

- `GET`、`PUT`、`DELETE` 必须符合幂等预期；`POST` 在支付、下单、审批、导入、消息发送等场景应有幂等键或业务唯一约束。
- 重试场景要明确重复请求返回的状态、响应体和冲突处理方式。
- 更新接口涉及并发覆盖时，应检查乐观锁、版本号、ETag/If-Match 或状态机约束。
- 异步任务创建要区分资源创建成功、任务已受理和任务执行结果，可使用 `202 Accepted` 和任务查询资源。

### 鉴权、授权和输入校验

- 所有非公开接口必须显式声明鉴权；公开接口要能从代码或文档看出是有意公开。
- 资源级授权要检查租户、组织、所有者、角色、数据权限和状态权限，不能只判断用户已登录。
- 输入校验应覆盖 body、query、path、header、文件上传和批量操作；复杂校验应在 DTO/schema/validator 中集中表达。
- 检查长度、格式、枚举、范围、必填、默认值、空白字符串、数组大小、文件类型、金额精度和时区。

### OpenAPI、文档和测试

- OpenAPI/Swagger 要覆盖路径、方法、参数、请求体、响应 schema、错误响应、安全方案、分页模型和示例。
- 文档中的状态码、错误码、字段必填、字段类型、枚举值、版本路径必须和实现一致。
- 新增或变更接口应有契约测试、Controller/Handler 测试、鉴权测试、输入校验测试、分页边界测试和兼容性回归测试。
- 对公开 API，建议检查 OpenAPI lint、schema diff、客户端生成或消费者契约测试是否存在。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。

## 搜索线索

### Codex

- 内容搜索优先使用 `rg -n`，文件发现优先使用 `rg --files`。
- 常用关键词：`@RequestMapping`、`@GetMapping`、`@PostMapping`、`Controller`、`Router`、`Route`、`handler`、`status`、`ResponseEntity`、`HttpStatus`、`throw new`、`errorCode`、`page`、`limit`、`sort`、`filter`、`OpenAPI`、`Swagger`、`SecurityRequirement`、`PreAuthorize`、`Authorize`、`AuthGuard`、`validate`。
- 搜索结果必须保留文件路径、行号和匹配代码片段，再读取具体文件上下文。

### Claude

- 使用 `Grep` 查内容、`Glob` 查文件、`Read` 读具体文件。
- 先定位路由和 Controller，再追 DTO/schema、错误处理、鉴权中间件、OpenAPI 文件和测试。

### Trae

- 使用 `SearchCodebase` 查接口入口和契约关键词，使用 `Read` 读取上下文。
- 需要跨文件判断时，先列出候选文件和证据，再做问题分级。

### 通用文件线索

- 路由/Controller：`*Controller*`、`*Resource*`、`*Handler*`、`routes.*`、`router.*`、`api/**`。
- DTO/schema：`*Request*`、`*Response*`、`*Dto*`、`schema.*`、`validator.*`、`serializers.*`。
- 文档：`openapi.yaml`、`openapi.json`、`swagger.*`、`api-docs/**`、注解式文档。
- 错误处理：`ExceptionHandler`、`ErrorCode`、`ApiError`、`ProblemDetail`、`writeError`、`HttpException`。
- 鉴权授权：`SecurityConfig`、`Permission`、`Policy`、`Guard`、`Interceptor`、`Middleware`、`PreAuthorize`。
- 测试：`*ControllerTest*`、`*ApiTest*`、`*ContractTest*`、`*.spec.*`、`*.test.*`。

## 输出格式

审查结果必须先列问题，按等级从高到低分组；没有问题的分组可以省略。

```markdown
## 阻断

- [文件路径:行号] 问题标题
  - 问题：具体违反的 API 设计点。
  - 影响：对安全、兼容性、客户端、性能或可维护性的影响。
  - 依据：引用当前代码证据和本 skill 的判断规则。
  - 建议：可执行的修复方向，不直接改代码。

## 高风险

## 中风险

## 建议优化

## 无法确认

- [文件路径:行号] 说明缺少哪些上下文，以及建议如何验证。

## 测试与文档缺口

- 说明缺失的契约测试、接口测试、OpenAPI 更新或兼容性验证。
```

如果没有发现问题，明确说明“未发现阻断/高风险/中风险问题”，并列出仍未覆盖的检查范围或测试缺口。

## 多平台兼容要求

- 不执行 Git 命令；用户明确要求 Git 操作时才使用对应平台工具。
- 不默认发起 HTTP 请求、数据库请求、构建、测试或格式化；需要执行时先说明目的和影响。
- 不修改代码；用户要求修复时，再按当前平台编辑工具和安全批量修改规则逐项处理。
- Windows 路径优先使用正斜杠；中文路径和中文内容读取、写入、复核必须显式按 UTF-8 处理。
- 输出中的本地文件引用必须包含可定位路径和行号；不使用只有文件名、没有行号的笼统描述。
