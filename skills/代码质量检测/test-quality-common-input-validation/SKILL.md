# 通用输入校验审查

## 定位

输入校验质量审查，重点判断外部输入是否在入口层、业务层和持久化层形成完整防线。

## 适用范围

- 用户要求审查请求参数、DTO、表单、接口入参、文件上传、ID 参数、批量操作参数或边界值时使用。
- 涉及 Controller/Handler/API、Service 业务规则、数据库约束、前后端校验一致性时使用。
- 只做代码层输入校验审查，不默认发请求、不跑 API 测试；需要动态验证时先输出建议。

## 检查流程

1. 找入口：Controller、Handler、路由、消息消费、定时任务、文件上传、外部回调、命令参数。
2. 列参数：必填字段、字符串、数值、集合、日期、文件、ID/引用、枚举、排序分页字段。
3. 跟数据流：从入口参数到 Service、DAO/Repository、SQL、文件路径、第三方请求和日志输出。
4. 检查三层防线：入口格式校验、业务合法性校验、数据库约束是否互相补位。
5. 每个问题必须给出文件路径、行号、缺失的校验项和建议补在哪里。

## 分级检查项

### 阻断

- **外部输入直接进入 SQL、命令或路径**：未校验的参数参与 SQL 拼接、系统命令、文件读写、下载路径或解压路径。
  - 怎么检查：追踪请求参数、文件名、排序字段、搜索关键字是否进入 SQL/命令/路径构造。
  - 修复方向：参数绑定、白名单、路径规范化、固定根目录限制。
- **ID/引用参数缺少权限归属校验**：只校验 ID 存在，不校验是否属于当前用户、租户、组织或角色权限范围。
  - 怎么检查：从 `id`、`userId`、`tenantId`、`orderId` 等参数追踪到查询和写操作条件。
- **批量操作无限制**：集合参数无最大数量限制，批量删除/更新/导入可一次处理过大数据集。
  - 怎么检查：查看 `ids`、列表、数组、文件行数、导入条数、分页大小是否限制。
- **文件上传缺少核心校验**：不限制大小、类型、文件名、目录，或只信任扩展名。
  - 怎么检查：查看上传入口是否校验 MIME、内容特征、文件大小、文件名清洗和存储目录。

### 高风险

- **必填和空白校验缺失**：`null`、空字符串、空集合、空对象进入业务逻辑。
  - 怎么检查：查看 DTO 注解、Schema、Pydantic、validator、手写校验和框架绑定配置。
- **字符串边界不足**：缺少长度、格式、枚举、字符白名单或富文本过滤。
  - 怎么检查：看用户名、手机号、邮箱、编码、备注、搜索词、URL、富文本字段的约束。
- **数值范围和精度不足**：金额、数量、折扣、页大小、比例、年龄、库存可为负数、过大或精度异常。
  - 怎么检查：确认最小值、最大值、小数位、单位和类型绑定。
- **日期时间范围不足**：开始时间晚于结束时间、未来生日、过早/过晚日期、时区不明确。
  - 怎么检查：查看日期格式、范围合理性、时间区间和前后端时区处理。
- **业务层校验缺失**：资源不存在、重复创建、状态不允许、库存不足、余额不足、重复提交未处理。
  - 怎么检查：在 Service 层确认存在性、唯一性、状态机、幂等和并发冲突处理。
- **只依赖前端校验**：后端没有独立校验或可被接口直接绕过。
  - 怎么检查：对比前端表单规则和后端 DTO/Service 校验，后端缺失的按风险标记。

### 中风险

- **数据库约束缺失**：业务必填字段无 NOT NULL，唯一字段无 UNIQUE，引用字段无外键或应用层一致性说明。
  - 怎么检查：查看 DDL、实体注解、迁移脚本和唯一性查询。
- **错误响应不清晰**：没有说明哪个字段错、期望格式是什么，或参数错误与业务冲突状态码混乱。
  - 怎么检查：查看校验异常处理、错误码、字段级错误集合和批量校验返回。
- **手动类型转换风险**：数字/日期先按字符串接收再手动解析，解析异常未处理。
  - 怎么检查：搜索 `parse`、`valueOf`、`convert`，查看框架类型绑定和异常分支。
- **排序分页字段未白名单**：`sort`、`orderBy`、`pageSize`、`limit` 未限制字段和范围。
  - 怎么检查：查看列表接口入参和动态 SQL/ORM 排序构造。
- **校验分散且重复**：同一字段规则在多个地方复制，容易不一致。
  - 怎么检查：搜索字段名、错误文案、正则和常量，判断是否应集中到 DTO、Schema 或校验器。

### 建议优化

- **边界测试缺口**：缺少正常值、最小/最大、越界、空值、特殊字符、类型错误、重复提交测试。
- **校验规则命名不清**：错误码、字段名、校验器名称无法表达业务含义。
- **批量校验体验差**：一次只返回第一个错误，用户需要多轮提交。
- **校验位置不合理**：格式校验应靠近入口，业务校验应在 Service，数据约束应落数据库。

## 判断标准

- 阻断：输入可直接造成注入、越权、任意文件访问、批量破坏或上传高危文件。
- 高风险：关键入参缺少必填、范围、格式、业务合法性或后端独立校验。
- 中风险：校验存在但边界不完整、错误响应不清晰、数据库约束不足或规则分散。
- 建议优化：当前可运行，但测试、可读性或校验体验可以提升。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。
- 不要求固定存在 `.claude`、`.codex`、`.trae` 等平台专属目录；按当前平台可用工具执行。

## 输出要求

按阻断、高风险、中风险、建议优化分组。每个问题使用以下格式：

```text
- [等级] 问题标题
  - 位置：path/to/file:line
  - 缺失校验：缺少必填、格式、范围、权限、状态或数据库约束
  - 原因：为什么这是风险
  - 建议：补在哪一层以及如何补
```

没有发现对应等级问题时写“未发现”。无法确认的问题要说明原因和下一步验证建议。

## 内置规则覆盖

- 校验层次：入口 DTO/Request 校验格式和范围，业务层校验状态和权限，数据库层用约束兜底唯一性和关系一致性。
- 必查字段：必填、长度、格式、枚举、数值范围、时间范围、文件大小/类型、分页大小、排序字段和 ID 归属。
- 信任边界：前端、网关、第三方回调、消息队列、定时任务、导入文件和内部接口都不能默认可信。
- 错误反馈：校验失败要返回可理解错误，但不能泄漏 SQL、堆栈、内部路径、规则细节或敏感数据。
- 绕过风险：批量接口、部分更新、默认值、空字符串、空数组、null、大小写和编码差异都要单独检查。

## 多平台兼容

- Codex：优先使用 `rg`、`rg --files`、`Get-Content -Encoding UTF8`，复杂任务使用 `update_plan`。
- Claude：优先使用 `Read`、`Grep`、`Glob`，复杂任务使用可用的 Todo 工具。
- Trae：优先使用 `SearchCodebase`、`Read`，复杂任务使用可用的任务清单工具。
- 任一平台都必须遵守：不默认执行 Git、不自动修复、不批量改文件、中文内容保持 UTF-8。
