# 通用性能质量审查

## 定位

代码层性能质量审查，重点识别查询、循环、缓存、并发、网络、内存、前端加载和渲染成本中的性能风险。

## 适用范围

- 用户要求审查“性能问题”“慢”“卡顿”“资源占用”“查询效率”“渲染性能”时使用。
- 涉及列表查询、导出、批处理、循环、缓存、远程调用、并发写、大文件、大对象、前端长列表和资源加载时使用。
- 只做静态代码审查，不默认运行压测、构建、接口请求或数据库执行计划；需要动态验证时先给出验证建议。

## 检查流程

1. 明确性能敏感路径：高频接口、列表页、导出导入、批处理、定时任务、渲染入口、文件处理。
2. 估算数据规模：输入集合大小、分页上限、表数据量、循环次数、远程调用次数、响应体大小。
3. 跟调用链：从入口到数据库、缓存、网络、文件、CPU 密集计算、内存分配和前端渲染。
4. 按分级检查项判断，每个问题必须包含文件路径、行号、触发条件和影响。
5. 无法确认真实耗时时，不下绝对结论，写明需要日志、指标、压测或执行计划验证。

## 分级检查项

### 阻断

- **无界数据处理**：列表、导出、批处理、消息消费没有分页、最大条数、批大小或超时限制。
  - 怎么检查：查看 `pageSize`、`limit`、导出数量、文件行数、批处理大小和默认上限。
- **循环内高成本调用**：循环中查库、RPC/HTTP、文件 IO、缓存网络请求或复杂解析。
  - 怎么检查：搜索循环结构，确认循环体内是否调用 DAO/Repository/HTTP client/文件读写。
- **可导致资源耗尽的实现**：一次性加载超大文件/数据集、无限递归、无限重试、无超时网络调用。
  - 怎么检查：查看读取方式、递归终止条件、重试策略、连接超时和读取超时。

### 高风险

- **N+1 查询**：先查列表，再逐项查询详情、权限、字典、关联数据。
  - 怎么检查：沿列表查询后的循环检查数据访问调用，判断是否可批量查询或 JOIN。
- **数据库查询性能风险**：`SELECT *`、缺少高频条件索引、索引失效、低效子查询、大偏移分页。
  - 怎么检查：查看 SQL、Mapper、ORM 查询、排序字段和 where 条件；无法确认索引时标记需验证。
- **缓存策略不当**：热点数据未缓存，缓存无 TTL，缓存击穿/穿透/雪崩无处理，更新后不失效。
  - 怎么检查：查看缓存 key、TTL、空值缓存、锁/互斥、刷新和失效逻辑。
- **重复计算或重复转换**：循环内重复 JSON 解析、正则编译、日期格式化、大对象映射、字符串拼接。
  - 怎么检查：搜索循环内 `parse`、`format`、`new`、`JSON`、正则和对象拷贝。
- **并发控制影响性能或正确性**：锁粒度过大、事务过长、同步阻塞、线程池/连接池配置不合理。
  - 怎么检查：查看锁范围、事务范围、线程池、连接池、队列长度和异步执行策略。
- **前端长列表渲染风险**：一次渲染大量节点、无虚拟滚动、频繁重渲染、搜索输入未防抖。
  - 怎么检查：查看列表数据量、组件循环、computed/watch、事件监听、滚动/resize/search 处理。

### 中风险

- **响应体过大**：接口返回多余字段、大对象、嵌套列表、前端只用少量字段。
  - 怎么检查：对照 DTO/VO 字段和页面实际使用字段，确认是否可瘦身。
- **资源释放不完整**：文件流、连接、订阅、定时器、事件监听未关闭或未取消。
  - 怎么检查：搜索 IO、stream、subscription、timer、listener，确认 finally/try-with-resources/生命周期清理。
- **算法复杂度偏高**：嵌套循环、重复搜索、列表内 `contains`、可用 Map/Set 却线性扫描。
  - 怎么检查：估算数据量和时间复杂度，重点看 O(n^2) 及以上路径。
- **网络请求冗余**：前端或服务端串行请求过多，可合并、并发或批量接口未使用。
  - 怎么检查：查看页面初始化、循环请求、服务编排和远程接口调用顺序。
- **资源加载未优化**：前端路由未懒加载、图片未压缩或懒加载、包体无拆分。
  - 怎么检查：查看动态 import、图片格式、组件按需加载、构建配置和资源引用。

### 建议优化

- **缺少性能指标**：关键路径没有耗时日志、慢查询日志、缓存命中率或前端性能埋点。
- **批处理可恢复性不足**：大任务没有分批、断点续跑、失败重试和进度记录。
- **对象复用不足**：高频路径频繁创建临时对象、正则、格式化器或缓冲区。
- **性能测试缺口**：缺少大数据量、并发、缓存失效、长列表、导出导入场景测试。

## 判断标准

- 阻断：可能造成资源耗尽、无限处理、接口不可用或明显拖垮数据库/服务。
- 高风险：高频路径或大数据场景下会显著变慢、放大成本或引入并发阻塞。
- 中风险：当前规模下可运行，但数据量增长后有明确性能退化点。
- 建议优化：能提升观测、批处理可靠性或长期性能余量。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。
- 不要求固定存在 `.claude`、`.codex`、`.trae` 等平台专属目录；按当前平台可用工具执行。

## 输出要求

按阻断、高风险、中风险、建议优化分组。每个问题使用以下格式：

```text
- [等级] 问题标题
  - 位置：path/to/file:line
  - 触发条件：什么数据量或调用场景下触发
  - 影响：耗时、资源、数据库、网络或渲染影响
  - 建议：如何优化或下一步验证
```

没有发现对应等级问题时写“未发现”。无法确认的问题要说明原因和下一步验证建议。

## 内置规则覆盖

- 数据访问：N+1、循环查询、无分页、重复聚合、无缓存的热点读取、事务内慢 IO 和不必要锁等待都是重点风险。
- 计算资源：嵌套循环、重复序列化、重复正则、无界集合、同步阻塞、未释放资源和大对象复制要结合数据量判断。
- 网络与外部调用：串行远程调用、无超时、无重试边界、无降级、请求放大和重复调用要说明触发场景。
- 前端/渲染：大列表无虚拟滚动、重复渲染、无稳定 key、大图无压缩、阻塞主线程和过量 watcher 都要定位。
- 证据要求：不能只凭“可能慢”下结论；需要给出触发条件、数据规模、调用频率或建议 profile/压测验证。

## 多平台兼容

- Codex：优先使用 `rg`、`rg --files`、`Get-Content -Encoding UTF8`，复杂任务使用 `update_plan`。
- Claude：优先使用 `Read`、`Grep`、`Glob`，复杂任务使用可用的 Todo 工具。
- Trae：优先使用 `SearchCodebase`、`Read`，复杂任务使用可用的任务清单工具。
- 任一平台都必须遵守：不默认执行 Git、不自动修复、不批量改文件、中文内容保持 UTF-8。
