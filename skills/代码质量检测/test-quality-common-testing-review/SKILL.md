# 通用测试质量审查

## 定位

测试质量审查，重点判断测试是否覆盖变更风险、关键业务、异常路径、边界输入、权限安全和回归场景。

## 适用范围

- 用户要求审查测试覆盖、单元测试、集成测试、回归测试、边界测试、权限测试或安全修复测试时使用。
- 新功能、缺陷修复、安全修复、权限变更、并发逻辑、关键业务逻辑变更后使用。
- 只做测试质量审查，不默认运行测试命令；需要运行测试或覆盖率命令时先给出建议并等待确认。

## 协作 skill

- `p3c-unit-testing`
- `test-p3c-code-quality`

## 检查流程

1. 识别变更风险：区分新功能、缺陷修复、安全修复、权限变更、数据一致性、并发和关键业务流程。
2. 建立覆盖映射：变更点 -> 测试文件 -> 单元/集成/E2E 类型 -> 断言内容。
3. 检查场景矩阵：正常值、异常值、边界值、空值、空集合、超长输入、特殊字符、类型错误、并发场景。
4. 检查覆盖率证据：有覆盖率报告时核对行覆盖、分支覆盖和关键逻辑覆盖；没有报告时只做静态判断并说明无法确认。
5. 检查测试质量：命名、AAA 结构、断言有效性、独立性、可重复性、外部依赖 Mock 和测试数据隔离。
6. 输出结论：每个问题必须包含文件路径、行号、缺失测试场景和建议补充方式。

## 分级检查项

### 阻断

- **关键风险完全没有回归测试**：安全、权限、资金、订单、库存、审批、删除、导入导出等修复缺少能复现风险的测试。
  - 怎么检查：搜索变更类、方法、接口、权限点对应测试，确认是否断言风险输入、越权角色或失败路径。
- **核心业务逻辑无直接测试**：关键计算、状态流转、幂等、并发写、事务回滚没有单元或集成测试。
  - 怎么检查：从 Service/Domain/核心函数定位测试文件，确认是否覆盖正常、异常和边界。
- **测试没有有效断言**：只验证“不报错”、只启动上下文、只调用方法但不校验结果。
  - 怎么检查：查看是否存在 `assert`、`expect`、`verify`，以及是否验证返回值、状态变更、异常、持久化结果或外部调用。
- **覆盖率证据显示关键模块低于底线**：有覆盖率报告且核心模块明显低于项目要求或关键分支未覆盖。
  - 怎么检查：读取现有 coverage 报告；没有报告时不要硬判覆盖率失败，只写“需覆盖率验证”。

### 高风险

- **正常/异常/边界流程缺失**：只测 happy path，没有最小值、最大值、越界、空值、空集合、异常返回。
  - 怎么检查：按每个入参建立场景矩阵，对照测试用例名称和断言。
- **接口或模块交互缺少集成测试**：Controller、Service、Repository、外部接口协作变更没有覆盖真实边界。
  - 怎么检查：搜索 API 测试、集成测试、数据库测试或 mock server 测试。
- **关键用户路径缺少 E2E 或等价流程测试**：登录、下单、支付、审批、导入导出等路径只测局部。
  - 怎么检查：查看 E2E、流程级测试或服务编排测试是否覆盖完整链路。
- **Mock 目标错误**：被测对象本身被 Mock，导致测试只验证 Mock 行为。
  - 怎么检查：检查 Mock 目标；外部依赖可 Mock，被测 service/component 不应被 Mock 掉。
- **安全输入和权限场景缺失**：XSS、SQL 注入样例、越权、未登录、角色不足、租户隔离没有测试。
  - 怎么检查：搜索特殊字符、权限角色、租户、登录态和安全异常断言。

### 中风险

- **测试不稳定**：依赖执行顺序、共享状态、固定等待、真实时间、随机值或外部网络。
  - 怎么检查：搜索 `sleep`、`delay`、`setTimeout`、`wait`、静态变量、共享缓存和真实外部依赖。
- **外部依赖隔离不足**：HTTP、数据库、文件系统、消息队列没有 Mock、测试容器、临时目录或专用测试数据。
  - 怎么检查：查看测试初始化、清理逻辑、测试 profile 和依赖替身。
- **测试数据准备重复且脆弱**：大量复制对象构造、魔法值、测试数据与业务规则耦合。
  - 怎么检查：查看 Arrange 段是否过长，是否可使用 Builder、Factory 或 fixture。
- **多个行为塞进一个测试**：一个测试覆盖多个无关分支，失败后定位困难。
  - 怎么检查：看测试名称与断言数量是否匹配同一个行为。
- **TDD 证据不完整**：新功能有测试但无法确认失败先行或覆盖范围。
  - 怎么检查：审查阶段通常无法确认 RED 阶段，只要求测试能表达需求、断言有效、覆盖关键路径。

### 建议优化

- **命名不表达场景和期望**：建议使用“被测方法_场景_期望结果”或项目已有等价规范。
- **AAA 结构不清晰**：Arrange、Act、Assert 混在一起，建议分段或提取辅助方法。
- **断言可读性不足**：断言只比较整体对象，失败信息不清晰。
- **覆盖报告未纳入流程**：建议在关键模块保留行覆盖、分支覆盖和排除范围说明。
- **测试类型比例不合理**：大量慢集成测试替代单元测试，或只有单元测试缺少关键流程测试。

## 判断标准

- 阻断：关键风险无回归测试、核心逻辑无直接测试、测试没有有效断言或覆盖证据显示关键模块严重不足。
- 高风险：主要业务、异常、边界、权限、安全、集成或 E2E 场景缺失。
- 中风险：测试可运行但稳定性、隔离性、数据准备、Mock 使用或可维护性存在明显问题。
- 建议优化：命名、结构、断言可读性、覆盖报告流程或测试类型比例可改进。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。
- 不要求固定存在 `.claude`、`.codex`、`.trae` 等平台专属目录；按当前平台可用工具执行。

## 输出要求

按阻断、高风险、中风险、建议优化分组。每个问题使用以下格式：

```text
- [等级] 问题标题
  - 位置：path/to/file:line
  - 缺失测试：缺少哪类测试或断言
  - 原因：为什么当前测试不足
  - 建议：补充什么测试、断言或验证证据
```

没有发现对应等级问题时写“未发现”。无法确认的问题要说明原因和下一步验证建议。

## 内置规则覆盖

- 测试层次：单元测试覆盖纯业务规则，集成测试覆盖数据库/消息/外部边界，端到端测试覆盖关键用户流程。
- 必测路径：成功、失败、边界、权限、并发、幂等、重试、超时、回滚、空数据和历史缺陷回归。
- 断言质量：断言要验证业务结果和副作用，不只验证“没有报错”；测试命名应表达场景和预期。
- 稳定性：避免真实时间、真实网络、固定临时路径、共享状态、顺序依赖和无界等待导致 flaky。
- E2E 质量：检查 Playwright/Cypress 等 E2E 是否覆盖关键用户路径，是否使用 Page Object 或等价封装、稳定 locator、明确断言和独立测试数据。
- E2E 稳定性：检查是否避免 `waitForTimeout`、固定 sleep 和动画竞态；失败时应保留 screenshot、trace、video 或 HTML report 便于定位。
- CI 与环境：检查 E2E baseURL、webServer、retry、worker、浏览器矩阵、artifact 上传、生产环境跳过高风险真实交易/支付流程。
- TDD 依据：新功能建议先写失败测试、最小实现、再重构；审查时只作为测试缺口判断，不强制自动运行。

## 多平台兼容

- Codex：优先使用 `rg`、`rg --files`、`Get-Content -Encoding UTF8`，复杂任务使用 `update_plan`。
- Claude：优先使用 `Read`、`Grep`、`Glob`，复杂任务使用可用的 Todo 工具。
- Trae：优先使用 `SearchCodebase`、`Read`，复杂任务使用可用的任务清单工具。
- 任一平台都必须遵守：不默认执行 Git、不自动修复、不批量改文件、中文内容保持 UTF-8。
