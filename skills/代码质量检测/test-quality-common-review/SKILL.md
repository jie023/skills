# 通用代码质量审查

## 定位

跨语言的代码层质量审查，重点判断可维护性、复杂度、重复代码、错误处理、日志、测试覆盖和结构职责问题。

## 适用范围

- 用户要求“检查代码质量”“审查模块”“看看有没有问题”且未限定具体语言专项时使用。
- 只做静态代码审查，不默认运行接口请求、测试命令或构建命令。
- 发现明确安全、数据库、性能、构建、语言框架专项问题时，可以在报告中建议追加对应专项 skill，但当前报告仍要给出已确认的问题。

## 检查流程

1. 明确审查范围：优先使用用户指定的文件、目录或模块；未指定范围时，只基于当前对话上下文，不主动使用 Git 推断变更。
2. 识别入口与调用链：先看外部输入、写操作、核心业务流程、异常分支、日志输出和测试文件。
3. 按分级检查项逐项判断：每个问题必须能落到具体文件路径和行号。
4. 无法确认的问题不要下结论，写明缺少的信息和下一步验证方式。
5. 输出报告，不自动修复；如用户要求修改，再按确认后的单项问题逐项处理。

## 分级检查项

### 阻断

- **安全红线**：硬编码密钥、SQL/命令/路径注入、XSS、越权、敏感信息泄露。
  - 怎么检查：搜索密钥、拼接 SQL/命令、直接输出用户输入、权限校验缺失、日志打印 token/密码/证件号。
  - 处理建议：标记阻断，并建议追加 `test-quality-common-security-review` 或相关安全测试。
- **数据丢失或错误写入风险**：无条件删除、批量更新缺少条件、事务边界错误、幂等性缺失。
  - 怎么检查：重点看 `delete`、`update`、批量导入、状态流转、事务提交/回滚和重复请求处理。
- **外部输入完全未校验**：请求参数、文件路径、第三方回调、消息队列数据直接进入业务或持久化层。
  - 怎么检查：从 Controller/API/任务入口追踪到 Service/DAO，确认空值、类型、长度、范围、枚举、权限上下文是否校验。
- **生产崩溃风险**：关键路径空指针、数组越界、未捕获异常、资源未释放、解析失败无兜底。
  - 怎么检查：查看空值来源、集合访问、类型转换、JSON/日期/金额解析、IO/连接释放和异常传播。

### 高风险

- **错误处理缺失或吞异常**：空 `catch`、只打印栈、不返回可处理错误、异常用于正常流程。
  - 怎么检查：搜索 `catch`、`throw`、`printStackTrace`、`console.error`、`logger.error`，判断是否保留上下文并给调用方明确结果。
- **复杂度超标**：单方法圈复杂度大于 10、嵌套深度大于 3、条件分支超过 5、链式调用过深。
  - 怎么检查：统计 `if/else/switch/case/for/while/catch/&&/||` 等决策点，观察是否可拆方法、策略、表驱动或提前返回。
- **职责混杂或 SOLID 违规**：Controller 写业务、Service 直接拼持久化细节、工具类包办多个领域。
  - 怎么检查：看类名、方法名和依赖方向是否一致；确认一个类是否有多个变化原因。
- **明显性能问题**：循环查库、无分页、重复远程调用、重复解析大对象、缓存策略错误。
  - 怎么检查：沿循环、DAO/Repository、HTTP/RPC、文件 IO、缓存读写路径确认调用次数和数据规模。
- **关键测试缺失**：核心分支、失败路径、边界值、回归场景没有测试。
  - 怎么检查：查找同模块测试文件，确认是否覆盖正常、异常、边界、权限或状态流转路径。

### 中风险

- **方法或文件过长**：单方法超过 50 行，单文件超过 800 行；200-400 行以上应关注职责是否膨胀。
  - 怎么检查：读取带行号内容，定位长方法边界和可拆分的业务步骤。
- **重复代码**：相同校验、转换、查询、日志、异常处理在多处复制。
  - 怎么检查：用关键业务字段、错误文案、SQL 片段、条件表达式搜索重复实现。
- **命名不清晰**：缩写、泛化名、布尔变量缺少 `is/has/can/should` 语义。
  - 怎么检查：看调用点是否需要反复跳转才能理解变量、方法、类的真实意图。
- **魔法值和硬编码配置**：业务阈值、状态码、超时、费率、环境差异写死在代码中。
  - 怎么检查：搜索数字、字符串常量、环境名、URL、状态字面量，判断是否应抽常量或配置。
- **依赖边界不清**：循环依赖、下层反向依赖上层、跨模块直接依赖实现类。
  - 怎么检查：查看 import、构造注入、模块路径和包层级，判断调用方向是否稳定。
- **日志不规范**：用 print 替代日志、日志级别错误、缺少关键上下文、重复刷屏。
  - 怎么检查：搜索 `print`、`console.log`、`System.out`、日志调用，判断级别、字段和异常栈是否合理。

### 建议优化

- **代码风格不一致**：import 排序、声明位置、格式、空行与项目现有风格不一致。
- **注释质量问题**：公共 API 缺少必要注释，复杂逻辑只写“做什么”不写“为什么”，存在过期注释。
- **TODO 不可追踪**：TODO 缺少负责人、日期或明确后续动作。
- **参数过多**：函数参数超过 5 个时，考虑参数对象或配置对象。
- **抽象不匹配**：多个策略分支适合策略模式；只有一个实现时避免为了抽象而抽象。
- **可测试性不足**：外部依赖不可替换、时间/随机/网络直接写死，导致单元测试难以隔离。

## 常用搜索线索

按当前平台工具执行搜索，结果必须包含文件路径、行号和具体命中内容。

```text
TODO|FIXME|HACK
printStackTrace|System.out|console.log|print\(
catch\s*\(|throw\s+new|except\s+
if\s*\(|else\s+if|switch\s*\(|case\s+
delete\s+|update\s+|truncate|drop\s+
password|secret|token|privateKey|accessKey
null|undefined|None|Optional|get\(
```

## 判断标准

- 阻断：可能导致安全事故、数据破坏、线上崩溃、关键输入未校验。
- 高风险：可能导致严重故障、明显维护成本上升、关键路径不可测或性能退化。
- 中风险：当前可运行，但后续修改容易引入缺陷或理解成本明显偏高。
- 建议优化：不影响当前行为，但能提升一致性、可读性或可测试性。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。
- 不要求固定存在 `.claude`、`.codex`、`.trae` 等平台专属目录；按当前平台可用工具执行。

## 输出要求

按阻断、高风险、中风险、建议优化分组。每个问题使用以下格式：

```text
- [等级] 问题标题
  - 位置：path/to/file:line
  - 原因：为什么这是问题
  - 建议：如何修改或下一步验证
```

没有发现对应等级问题时写“未发现”。无法确认的问题要说明原因和下一步验证建议。

## 内置规则覆盖

- 通用审查维度：正确性、安全、可维护性、性能、错误处理、测试覆盖和项目规范一致性。
- 代码风格：命名表达意图，函数单一职责，参数数量受控，嵌套深度可读，避免魔法值和无用注释。
- 架构模式：分层边界清晰，依赖方向稳定，业务规则不散落在控制器、UI、脚本或基础设施代码里。
- 坏味道：重复逻辑、过度抽象、长函数、全局可变状态、隐式副作用、吞异常和调试残留都需要定位到具体证据。
- 审查纪律：只报告代码证据能支撑的问题；无法确认时给验证建议，不把风格偏好伪装成缺陷。

## 多平台兼容

- Codex：优先使用 `rg`、`rg --files`、`Get-Content -Encoding UTF8`，复杂任务使用 `update_plan`。
- Claude：优先使用 `Read`、`Grep`、`Glob`，复杂任务使用可用的 Todo 工具。
- Trae：优先使用 `SearchCodebase`、`Read`，复杂任务使用可用的任务清单工具。
- 任一平台都必须遵守：不默认执行 Git、不自动修复、不批量改文件、中文内容保持 UTF-8。
