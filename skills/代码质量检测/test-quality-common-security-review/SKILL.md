# 通用代码安全审查

## 定位

代码层安全质量审查，重点识别硬编码密钥、输入校验缺失、注入、XSS、鉴权绕过、敏感信息泄露、错误信息泄露和依赖安全风险。

## 适用范围

- 用户要求“检查安全问题”“安全审查”“看看有没有漏洞”时使用。
- 修改或审查登录、鉴权、权限、支付、数据访问、文件上传、外部回调、第三方接口、富文本、日志、错误处理、依赖版本时使用。
- 只做防御性代码审查，不生成攻击脚本，不尝试绕过系统，不执行真实攻击流量。
- 发现疑似泄露的真实密钥时，报告风险并建议轮换密钥，不在回复中重复完整密钥内容。

## 协作 skill

- `secure-code-checklist`
- `p3c-security-rules`

## 检查流程

1. 明确审查范围：优先使用用户指定文件、目录或模块；未指定时按当前上下文审查，不主动使用 Git 推断变更。
2. 找入口：Controller/API、路由、定时任务、消息消费、文件上传、Webhook、前端输入、后台管理操作。
3. 跟数据流：从外部输入追踪到校验、鉴权、业务处理、SQL/命令/文件/网络调用、日志和响应。
4. 按分级检查项逐项判断，每个问题必须包含文件路径、行号、影响和修复建议。
5. 对无法确认的问题写明缺少的信息和下一步验证方式；不自动修复。

## 分级检查项

### 阻断

- **硬编码密钥或凭证**：API Key、密码、Token、连接串、私钥、证书口令写在代码、配置样例或前端包中。
  - 怎么检查：搜索 `password`、`secret`、`token`、`apiKey`、`accessKey`、`privateKey`、`jdbc:`、`.pem`、`.p12`、`.env`。
  - 判断标准：真实凭证或疑似可用凭证为阻断；前端环境变量中包含敏感值也按泄露处理。
- **鉴权或授权缺失**：敏感接口没有登录校验、角色校验、资源归属校验或越权防护。
  - 怎么检查：从路由/Controller 查权限注解、中间件、拦截器和 Service 层资源 owner 校验。
  - 重点场景：后台管理、订单、资金、用户资料、导出、删除、审批、状态变更。
- **高危注入**：SQL/命令/LDAP/XPath/模板表达式由外部输入拼接。
  - 怎么检查：追踪用户输入是否进入 SQL 字符串、shell 命令、脚本解释器、模板渲染、动态表达式。
  - 修复方向：参数化查询、白名单字段、参数数组、禁止 shell 字符串拼接。
- **路径遍历或任意文件访问**：用户控制文件名、路径、下载参数、压缩包解压路径且未规范化限制。
  - 怎么检查：搜索文件上传、下载、解压、读取、删除逻辑，确认是否规范化路径并限制在指定目录。
- **敏感信息直接泄露**：响应体、异常信息、日志或前端页面暴露密码、Token、身份证、手机号、内部表名、堆栈。
  - 怎么检查：查看异常返回、全局异常处理、日志参数、DTO/VO 字段和序列化输出。

### 高风险

- **输入校验不完整**：外部输入缺少类型、长度、范围、枚举、格式、文件类型、大小或业务权限校验。
  - 怎么检查：从入口参数追踪到业务使用点，确认校验发生在可信边界内且不能被绕过。
- **XSS 风险**：未转义输出、富文本未白名单过滤、危险 HTML 注入、URL scheme 未限制。
  - 怎么检查：前端关注 `v-html`、`dangerouslySetInnerHTML`、`innerHTML`；后端关注富文本存储和模板输出。
  - 修复方向：默认转义、富文本白名单、URL 只允许 `http/https`、启用 CSP。
- **CSRF 或会话防护缺失**：Cookie 会话接口未启用 CSRF 防护，Cookie 缺少 `HttpOnly`、`Secure`、`SameSite`。
  - 怎么检查：查看安全框架配置、Cookie 设置、跨站请求策略和敏感写接口。
- **JWT/Session 校验薄弱**：未验签、未验过期、未验 issuer/audience、刷新逻辑不安全、退出后仍可用。
  - 怎么检查：查看 token 解析、签名算法、过期时间、刷新令牌存储和服务端失效策略。
- **公网敏感接口缺少限流**：登录、注册、验证码、找回密码、导出、搜索等接口无限制。
  - 怎么检查：查看网关、中间件、注解、过滤器或服务层限流逻辑，确认触发后返回明确错误。
- **依赖存在明显安全风险**：使用已知高危版本、废弃库、过度依赖或从不可信源加载代码。
  - 怎么检查：查看依赖清单、锁文件和安全公告；不能联网时标记为需进一步扫描确认。

### 中风险

- **错误处理暴露内部细节**：用户看到表名、SQL、类名、堆栈、服务器路径或三方错误原文。
  - 怎么检查：查看全局异常处理和接口返回包装，区分用户消息和内部日志。
- **日志上下文或脱敏不足**：日志缺少业务上下文，或直接输出请求体、响应体、身份证、手机号、银行卡、Token。
  - 怎么检查：搜索日志调用，确认敏感字段脱敏，异常日志保留必要上下文和堆栈。
- **配置隔离不足**：不同环境共用密钥、回调地址、白名单、CORS、调试开关或管理端配置。
  - 怎么检查：查看配置文件、环境变量读取、默认值和环境分支。
- **文件上传校验不足**：只校验扩展名，未限制大小、MIME、内容特征、存储目录和访问权限。
  - 怎么检查：查看上传入口、文件名生成、存储路径、访问 URL、病毒扫描或异步处理策略。
- **SSRF 风险**：服务端请求用户可控 URL，未限制协议、域名、内网地址和重定向。
  - 怎么检查：搜索 HTTP 客户端调用，追踪 URL 是否来自请求参数或数据库可控字段。

### 建议优化

- **最小权限不足**：数据库账号、云权限、接口权限过大，建议按业务最小权限拆分。
- **安全响应缺少闭环**：发现泄露后没有轮换、审计、相似问题排查和影响面说明。
- **安全测试缺口**：关键权限、输入校验、文件上传、异常返回没有单元或集成测试覆盖。
- **安全配置可读性差**：规则分散、默认值不明确、白名单来源不清楚。

## 常用搜索线索

按当前平台工具执行搜索，结果必须包含文件路径、行号和具体命中内容。不要把疑似密钥完整复制到最终报告中。


## 判断标准

- 阻断：可能导致真实凭证泄露、越权、数据泄露、远程命令执行、任意文件读写或高危注入。
- 高风险：可能导致账号风险、敏感接口被滥用、XSS、CSRF、限流缺失或安全边界被绕过。
- 中风险：当前影响受限，但会扩大攻击面、泄露内部细节或降低排障安全性。
- 建议优化：不构成立即漏洞，但能提升安全基线、审计能力和长期可维护性。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。
- 不要求固定存在 `.claude`、`.codex`、`.trae` 等平台专属目录；按当前平台可用工具执行。

## 输出要求

按阻断、高风险、中风险、建议优化分组。每个问题使用以下格式：

```text
- [等级] 问题标题
  - 位置：path/to/file:line
  - 影响：可能造成什么后果
  - 原因：为什么存在风险
  - 建议：如何修复或下一步验证
```

没有发现对应等级问题时写“未发现”。无法确认的问题要说明原因和下一步验证建议。

## 内置规则覆盖

- 高危安全项：认证绕过、越权访问、SQL/命令/模板注入、XSS、SSRF、路径穿越、反序列化、任意文件读写。
- 敏感信息：密钥、token、密码、连接串、测试账号、内部地址不得写入代码、日志、前端产物或错误响应。
- 密钥管理：生产密钥必须来自环境变量或密钥管理服务，不同环境隔离；疑似泄露时要建议轮换、审计和相似问题排查。
- 权限判断：必须验证租户、用户、角色、资源归属和对象级权限；不能只依赖前端隐藏或接口参数可信。
- 输入输出：所有外部输入要做类型、范围、格式、长度、枚举、文件类型和编码校验；输出到 HTML、SQL、Shell、路径、日志前要按上下文处理。
- 安全报告：只暴露最小必要代码片段，不在报告里完整复述密钥；疑似密钥要脱敏并建议轮换。

## 多平台兼容

- Codex：优先使用 `rg`、`rg --files`、`Get-Content -Encoding UTF8`，复杂任务使用 `update_plan`。
- Claude：优先使用 `Read`、`Grep`、`Glob`，复杂任务使用可用的 Todo 工具。
- Trae：优先使用 `SearchCodebase`、`Read`，复杂任务使用可用的任务清单工具。
- 任一平台都必须遵守：不默认执行 Git、不自动修复、不批量改文件、中文内容保持 UTF-8。
