# 通用数据库访问质量审查

## 定位

数据库访问层质量审查，重点识别 SQL 安全、查询性能、索引、事务、数据完整性、迁移和租户权限过滤问题。

## 适用范围

- 用户要求审查 SQL、Mapper、Repository、DAO、数据访问层、数据库性能、事务、迁移或表结构时使用。
- 涉及列表查询、批量更新、删除、导入导出、租户数据、权限过滤、索引、分页和事务边界时使用。
- 只做静态代码和配置审查，不默认连接数据库、不执行 SQL、不跑 EXPLAIN；需要执行时必须先取得用户确认。

## 协作 skill

- `p3c-mysql-database`

## 检查流程

1. 明确范围：优先使用用户指定的 SQL、Mapper、DAO、Repository、Migration、Schema 文件。
2. 找数据入口：Controller/API、Service 方法、定时任务、消息消费、导入任务和后台操作。
3. 跟查询和写入路径：从参数到 SQL 构造、条件过滤、分页、事务、异常处理和返回结果。
4. 按分级检查项判断，每个问题必须落到文件路径和行号。
5. 无法确认索引或执行计划时，不臆测结论，写明需要表结构、索引或 EXPLAIN 进一步验证。

## 分级检查项

### 阻断

- **SQL 注入风险**：使用字符串拼接构造 SQL、排序字段、表名、列名、`IN` 条件或动态片段，且输入可被外部控制。
  - 怎么检查：搜索 `${}`、字符串 `+` 拼 SQL、`createStatement`、`rawQuery`、`ORDER BY` 动态字段。
  - 修复方向：参数化查询、ORM 参数绑定、白名单校验动态字段。
- **越权或租户过滤缺失**：查询、更新、删除没有用户归属、租户、组织、权限条件。
  - 怎么检查：从接口身份上下文追踪 SQL 条件，确认 `tenant_id`、`user_id`、`org_id` 或资源 owner 校验。
- **高危批量写操作**：删除或更新缺少 WHERE、条件过宽、无数量限制、无二次确认或无幂等保护。
  - 怎么检查：搜索 `delete`、`update`、批量导入、状态批改，确认条件、上限、事务和回滚策略。
- **敏感数据明文存储**：密码、Token、证件号、银行卡、密钥等直接入库或查询返回。
  - 怎么检查：查看字段名、DTO/VO 映射、加密/脱敏工具、日志和导出逻辑。

### 高风险

- **大表查询无分页或无限制导出**：列表接口、后台查询、导出接口没有默认分页和最大页大小。
  - 怎么检查：查看查询入参、分页对象、`limit`、游标、导出批处理和最大条数限制。
- **N+1 查询或循环 SQL**：循环内调用 DAO/Mapper/Repository 或远程查询。
  - 怎么检查：搜索循环内的数据访问调用，确认是否可批量查询、JOIN、缓存或一次性加载。
- **索引缺失或索引失效**：高频 WHERE/JOIN/ORDER BY 字段无索引，或函数包裹、隐式类型转换、`LIKE '%xxx'` 导致索引失效。
  - 怎么检查：结合查询条件和表结构判断；没有表结构时标记“需索引信息验证”。
- **事务边界不合理**：事务过大、事务内 RPC/HTTP/消息发送、长时间 IO、异常未触发回滚。
  - 怎么检查：查看事务注解/声明式事务范围、外部调用位置、异常捕获和回滚规则。
- **并发写一致性风险**：库存、余额、状态流转、审批等缺少乐观锁、悲观锁、唯一约束或幂等键。
  - 怎么检查：查看更新条件是否包含版本/状态，是否有唯一索引、幂等记录和重复提交处理。
- **表结构关键约束缺失**：业务唯一字段没有唯一约束，必填字段可空，金额精度不明确。
  - 怎么检查：查看 DDL、实体注解或迁移脚本，确认 NOT NULL、唯一约束、字段类型和精度。

### 中风险

- **`SELECT *` 或返回字段过多**：查询不限定字段，导致带宽、序列化和敏感字段暴露风险。
  - 怎么检查：搜索 `select *`、实体全量映射和接口响应对象。
- **索引设计不合理**：冗余索引、低选择性字段索引过多、联合索引顺序不符合最左前缀。
  - 怎么检查：结合查询条件顺序、排序字段和更新频率判断；每表索引过多需要提示写入成本。
- **软删除/硬删除策略不清**：业务需要审计或恢复却直接物理删除，或软删除查询漏过滤。
  - 怎么检查：查看删除接口、查询条件、审计字段和数据恢复要求。
- **迁移脚本不可回滚或大表 DDL 风险**：无回滚脚本、无备份说明、大表同步 DDL、数据迁移不分批。
  - 怎么检查：查看 migration 文件、批处理大小、失败重试、断点续跑和备份方案。
- **连接池和超时配置不明确**：数据源连接池、查询超时、事务超时缺省不清。
  - 怎么检查：查看数据源配置、ORM 超时配置和慢查询处理策略。

### 建议优化

- **审计字段不完整**：缺少创建人、创建时间、更新人、更新时间、删除标记等必要字段。
- **DAO 职责不清**：数据访问层掺入业务规则，或 Service 直接拼复杂 SQL。
- **读写分离和只读事务不足**：纯查询没有只读事务或读库路由标识。
- **SQL 可读性差**：动态 SQL 条件过多、命名不清、重复片段未抽取。
- **数据测试缺口**：缺少分页边界、空结果、大数据量、并发写、重复提交和迁移回滚测试。

## 判断标准

- 阻断：可能导致注入、越权、数据破坏、敏感数据泄露或不可逆批量写事故。
- 高风险：可能导致大表拖垮、数据不一致、事务阻塞、并发错误或关键约束缺失。
- 中风险：当前可运行，但性能、迁移、索引或可维护性存在明显隐患。
- 建议优化：不直接影响正确性，但可提升审计、可读性、测试性和长期演进能力。

## 执行规则

- 只做代码层质量审查，不跑 API 请求。
- 不默认执行任何 Git 操作。
- 不自动修复问题，只输出问题和修复建议。
- 不照搬强制调用、自动修复、hooks、后台观察逻辑。
- 输出必须包含文件路径、行号、问题等级、原因、建议。
- 报告输出遵循 `test-quality-report-template`。
- 如需要修改代码，必须等待用户确认后逐项处理。
- 多文件修改必须按安全批量修改规则执行。
- 中文路径和中文内容必须保持 UTF-8。
- 不要求固定存在 `.claude`、`.codex`、`.trae` 等平台专属目录；按当前平台可用工具执行。

## 输出要求

按阻断、高风险、中风险、建议优化分组。每个问题使用以下格式：

```text
- [等级] 问题标题
  - 位置：path/to/file:line
  - 影响：安全、性能、数据一致性或可维护性影响
  - 原因：为什么存在问题
  - 建议：如何修复或下一步验证
```

没有发现对应等级问题时写“未发现”。无法确认的问题要说明原因和下一步验证建议。

## 内置规则覆盖

- SQL 安全：动态 SQL 必须参数化；排序字段、表名、列名等无法参数化的位置要做白名单。
- 性能风险：N+1 查询、循环内访问数据库、缺分页、缺索引、全表扫描、模糊前缀、过大 IN、无界批量更新都要重点检查。
- 事务一致性：跨表写入、库存/余额/状态流转、消息发送和缓存更新要有明确事务边界和失败补偿。
- 数据模型：字段类型、长度、默认值、唯一约束、外键/逻辑关联、软删除和审计字段要与业务语义一致。
- ORM/Mapper：避免把复杂业务拼在 SQL 字符串里；Repository/Mapper 返回模型不能泄漏不该暴露的数据。

## 多平台兼容

- Codex：优先使用 `rg`、`rg --files`、`Get-Content -Encoding UTF8`，复杂任务使用 `update_plan`。
- Claude：优先使用 `Read`、`Grep`、`Glob`，复杂任务使用可用的 Todo 工具。
- Trae：优先使用 `SearchCodebase`、`Read`，复杂任务使用可用的任务清单工具。
- 任一平台都必须遵守：不默认执行 Git、不自动修复、不批量改文件、中文内容保持 UTF-8。
